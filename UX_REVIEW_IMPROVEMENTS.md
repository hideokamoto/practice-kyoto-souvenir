# きょう再見 UI/デザイン リニューアル提案

Apple Human Interface Guidelines（HIG, 2025-06 Liquid Glass 改訂版）を単一ソースとする `apple-hig-design` スキルの
7 フェーズ・ワークフローに沿って現状の UI を監査し、リニューアル案をまとめた。

## 0. 適用範囲の前提（最初に明記すること）

本アプリは Ionic Angular + Capacitor（`capacitor.config.ts` で iOS/Android バンドル対象、`appId: jp.kyoto_saiken.app`）。
描画は HTML/CSS/WebView であり、UIKit/SwiftUI のネイティブレンダリングではない。したがって：

- HIG を**そのまま規範**にはできない。`references/web-adaptation.md` の翻訳ルールに従う
- Liquid Glass の模写・SF Symbols・San Francisco フォントの移植は**禁則**（ライセンス上 Apple プラットフォーム限定のため）。現状すでにこの原則は守られている（Ionicons 使用、`-apple-system` を含むシステムフォントスタック使用、`prefers-color-scheme` 対応）→ ここは維持する
- 一方で Ionic の `ios` モードは「ネイティブに近い挙動（Large Title の collapse、スワイプ戻る等）」を提供しており、Capacitor 経由で実機 iOS 上ではこれが活きる。この構造自体は正しく、崩さない

つまり本提案は「Apple 風の見た目にする」提案ではなく、**HIG が体現している設計原則（階層・一貫性・段階的開示・アクセシビリティ）を Web/ハイブリッドアプリの語彙に翻訳して適用する**提案である。

現状の視覚言語（朱色 `#b6402c` ＋ 和紙背景 ＋ 明朝体見出しの「エディトリアル和風」デザイン、`src/global.scss` コメントに "きょう再見 Ionic Redesign v2" とある）は独自ブランドとして完成度が高く、これ自体を Apple 的な意匠に置き換えることは提案しない。今回の主眼は **「v2 デザインが Discover/Favorites/Plans には適用されたが、詳細ページ（sight-detail / souvenir-detail）に適用されずシステムが分裂している」構造的な一貫性の欠落を直す**ことと、HIG 監査観点での個別の改善である。

## 2. Phase 2: 構造とナビゲーション

### 2.1 タブバー構成 — 良好

`tabs.page.html` の 発見 / お気に入り / プラン の3区分は HIG の「タブバーはアプリのトップレベル区分、常に可視、区分数は少なく」に合致。「このアプリについて」を各画面ヘッダーのボタンに逃がしているのも適切（タブに押し込んでいない）。**維持**。

### 2.2 Discover 画面の情報階層が DESIGN.md の設計意図と実装でズレている（最重要指摘）

`DESIGN.md` は明確に「レベル1（唯一の必須）: 今週末のおすすめ」「画面の中央、最大の視覚的強調」と定義し、統計情報・ランダムプラン作成などは意図的に削除したと書かれている。

しかし `discover.page.html` の実装は、

1. 「おすすめスポット」ヒーローカード（観光地）
2. 「今日のピックアップ」ヒーローカード（お土産）

の**2 つの同格ヒーローカード**を縦に並べている。HIG Layout の「重要情報を最初に（読み順で最重要項目を上に）」は「1 つの主役」を前提にした原則であり、同じ視覚的重みのカードが 2 つ並ぶと「今日、何をしよう？」という単一の問いに対する答えが 2 つ提示されることになり、DESIGN.md 自身が禁じた「選択肢過多」の状態に戻ってしまっている。

**提案**:
- 観光地とお土産を対等に扱うのをやめ、「メイン提案（1件・大）」＋「関連するお土産（1件・小さいインラインカードか、メインカード内のサブセクション）」の非対称構成にする
- あるいはセグメント（観光地 / お土産）で "今日の1件" 自体を切り替えられるようにし、常に画面には 1 つのヒーローだけを出す
- どちらを採るにせよ、DESIGN.md を実装のソース・オブ・トゥルースとして再度同期させる（ドキュメントと実装が乖離した状態は放置しない）

### 2.3 フィルターチップが機能していない（バグ相当・要修正）

`discover.page.html` の `.chip-row` には「すべて / 寺社 / 庭園 / 未訪問 / お気に入り」の5チップがあるが、`SightsService`（`sights.service.ts`）が持つ `Sight` 型に **`category`/`type` に相当するフィールドが存在しない**。`setFilter()` は `activeFilter` を更新して `getRandomSuggestions()` を呼ぶだけで、`selectSightFromPool()` は `activeFilter` を一切参照していない。

→ 「寺社」「庭園」チップは**タップしても何も起きない飾り**になっている。HIG Patterns の「フィードバック: 現在の状態を正しく伝える」に反する（ユーザーは選んだつもりで何も変わらない）。

**提案**: 以下のいずれかを選択する
- (A) データセット側（`kyoto-sights.json` 生成元の CSV）にカテゴリ列を追加し、実際にフィルタする
- (B) 実装できないなら、少なくとも「未訪問」「お気に入り」（これは `userDataService` から実データがあり実装可能）だけを残し、「寺社」「庭園」は次回リリースまで**画面から外す**

見た目だけのコントロールを残すことは、HIG が最も嫌う「約束したのに応答しない UI」である。

### 2.4 検索の位置づけ

現状、検索（`app-search-sights` / `app-search-souvenirs`）は「探索モードを開く → セグメントを選ぶ → 検索する」という 3 手順の奥にある。DESIGN.md の意図（探索は能動的な人だけが辿る二次動線）とは整合しているため、これは**問題ではなく意図通り**と判断する。ただし `favorites.page.html` の検索バーは常時表示で一次導線になっており、Discover とは非対称。一貫性の観点で「検索の格」を画面ごとに揃えるか、揃わない理由（Discover は意図的に検索を隠す設計）を明文化しておくとよい。

## 3. Phase 3: 視覚基盤

### 3.1 タイポグラフィ — スタイルトークン化されていない

HIG は「生のポイント指定でなくテキストスタイルを使う」ことで Dynamic Type 相当の一括制御を可能にすることを求める。現状はテンプレート内に `style="font-size:13px;color:var(--ion-color-step-500)"` のようなインラインスタイルが散在しており（`discover.page.html` L53, L95, L133 ほか多数）、SCSS 側の `.hero-card .reading`（12px）や `.greeting-block .greeting-sub`（13px）と**微妙にサイズが異なる同種テキスト**が複数存在する。

**提案**: `global.scss` に和文編集用のタイプスケールを定義する（例）

```scss
:root {
  --step-eyebrow: 11px;   /* ラベル・オーバーライン */
  --step-caption: 12px;   /* 補助情報 */
  --step-body-sm: 13px;   /* 小さめ本文 */
  --step-body: 14px;      /* 標準本文 */
  --step-subhead: 15px;
  --step-headline: 17px;  /* iOS Body/Headline 相当 */
  --step-title: 22px;     /* カード見出し */
  --step-display: 32px;   /* Large Title */
}
```

インラインスタイルを撤去し、上記トークン＋クラスに置き換える。rem ではなく px 起点なのは Ionic 既定に合わせているためだが、ブラウザのテキストサイズ拡大設定（200%）で崩れないかは Phase 7 で別途検証が必要。

### 3.2 カラー — おおむね良好、コントラスト要検証

朱色ベースのライト/ダーク両対応カラートークンが `variables.scss` に定義済みで、HIG の「システムカラー優先・セマンティックトークン化」の思想を Web に正しく翻訳できている。良好な設計。

一方で以下は未検証のため Phase 7 の監査対象とする:
- `.hero-card .reading`（12px, `--ion-color-step-500` on `--ion-card-background`）— 17px 未満は 4.5:1 必須
- `.chip-on-color`（11px, `--ion-color-step-600` on 透明背景）
- `sight-detail.page.scss` の `--ion-color-medium, #666` のようなフォールバック値（実運用では CSS 変数が必ず定義されているため到達しないはずだが、フォールバック自体が実テーマと無関係な色でありメンテナンス上のノイズ）

### 3.3 マテリアル（Liquid Glass の「構造だけ移植」の適用余地）

HIG 2025 改訂の核は「コントロール層とコンテンツ層の分離」であり、模様としての磨りガラスではなく**構造原理**として移す対象、と `web-adaptation.md` は明記している。現状 `ion-header [translucent]="true"` は使われているが、コンテンツ側のカード（`.hero-card`）はコントロール層と同じく不透明・フラットで、層の区別が視覚的に弱い。

**提案（任意・低優先）**: ツールバー/タブバーにのみ薄い `backdrop-filter: saturate(180%) blur(8px)` 相当の半透明を与え、カード（コンテンツ層）は現状どおり不透明に保つことで「浮いている操作層」と「置かれているコンテンツ」の区別を強める。ただし `prefers-reduced-transparency` に相当する配慮（Web には直接の media query がないため、常時トグルできる設定を用意するか、控えめな透明度に留める）とセットで行うこと。

## 4. Phase 4: コンポーネント選定

### 4.1 詳細ページ間の実装レベル格差（構造的な最重要課題）

`sight-detail.page.html/scss` と `souvenir-detail.page.html` を比較すると、同じ「詳細ページ」でありながら実装レベルがまったく異なる。

| 要素 | sight-detail | souvenir-detail |
|---|---|---|
| エディトリアルデザイン（明朝体・朱色トークン） | 未適用（generic な `border-radius: 8px/12px`, `--ion-color-primary, #3880ff` という**現テーマと無関係な青のフォールバック**が残存） | 未適用 |
| 写真 | あり（`.photo-container`） | **なし** |
| 構造化された情報（アクセス・営業・料金・アクセシビリティ・備考） | `ion-item-divider` で整理済み | **本文をそのまま流し込むだけ**（`{{souvenir.description}}` 1行） |
| お気に入り/訪問済みボタン | ヘッダーアイコン＋本文内の大きいアクションボタン2種 | **ヘッダーアイコンのみ**（本文内ボタンなし） |
| 「プランに追加」ボタン | あり | **なし** |
| 前後ナビゲーション | あり | なし |

sight-detail は明らかに新しい「v2」の対象として作り込まれ、souvenir-detail は旧仕様のまま取り残されている。Discover 画面には「今日のピックアップ（お土産）」という同格の主要導線があるのに、その着地点である souvenir-detail が明らかに手薄というのは、**ユーザー体験としてもっとも致命的な一貫性の欠落**である（HIG Foundations「アプリ内全体でスタイル・詳細度を統一する」に反する）。

**提案（P0）**: souvenir-detail を sight-detail と同じコンポーネントパターンに揃える
- `hero-card` 系のトークン・明朝体見出しを適用
- 写真（データがあれば）、価格・購入可能場所などお土産固有情報の構造化表示
- 本文内アクションボタン（お気に入り／訪問済み／プランに追加）を追加
- 可能なら sight-detail 側とアクションボタンエリアの Angular コンポーネントを共通化し（例: `<app-detail-actions>`）、今後 3 つ目のコンテンツタイプが増えても分岐が発散しないようにする

### 4.2 FAB（Floating Action Button）の妥当性

Discover のシャッフル FAB、Plans の追加 FAB はいずれも Material Design 由来のパターンで、HIG（iOS）には存在しないコンポーネントである。`web-adaptation.md` は「OS 部品の見た目コピーを盲信しない」と同時に「機能要件（主要アクションの到達性）に翻訳する」ことを求めている。

- Discover のシャッフル FAB は、ヒーローカード自体がタップ可能でカード内に既にボタンがあるため、**「別の提案」という機能がカード内ボタンと FAB の2箇所に重複**している（実際 `getRandomSuggestions()` は FAB からしか呼べず、カード内には「別の提案」ボタンが見当たらない＝ DESIGN.md にある「別の提案」ボタンが実装から消えている可能性が高い）。ここも DESIGN.md と実装の乖離。
- Plans の追加 FAB は空状態時のボタンと機能重複するが、リストが埋まった状態での主要導線として妥当（一貫してアクセス可能な「今すぐプランを作る」動線として機能要件を満たす）。**維持でよい**。

**提案**: Discover 側は FAB を残すなら「別の提案」の唯一の入口として意味づけを明確にし、ラベル（またはツールチップ的な初回説明）を用意する。あるいは HIG 的に素直な「ヘッダーの操作アイコン」に統合してもよい。

### 4.3 セグメンテッドコントロールの用途 — 適切

Discover の探索モード内（観光地/お土産）、Favorites（すべて/観光地/お土産）、Plans（これから/完了）はいずれも「画面内のビュー切替」であり、HIG の「ナビゲーション（画面遷移）には使わない」というセグメント/タブバーの使い分け規則に正しく従っている。**維持**。

## 5. Phase 5: パターンとインタラクション

### 5.1 削除操作にアラート／Undo がない

`favorites.page.html` / `plans.page.html` の `ion-item-option color="danger" (click)="removeFavorite(item)"` は確認なしの即時削除。HIG は「頻繁で取り消し可能な削除には確認アラートを乱用しない」としているため確認ダイアログなし自体は正しい判断だが、**取り消せないなら Undo が実質ない**（"取り消し可能" の前提が満たされていない）。

**提案**: 削除後に「元に戻す」付きのトースト（`ionic`の `ToastController` + `buttons` で Undo アクション）を出す。これは HIG Patterns の「取り消し可能な操作」原則を Web/Ionic の語彙で満たす標準的な手段。

### 5.2 ローディング — 概ね良好

Discover はスケルトン（`app-list-item-skeleton`）とスピナーを併用し、「何かをすぐ見せる」という HIG の loading 原則に近い。ただし初回ロード時にスピナーのみ（スケルトンなし）の分岐があるため、リスト部分もスケルトンに統一すると一貫する。

### 5.3 モーション — Reduce Motion 未対応

`global.scss` 内 `.exploration-toggle { transition: background 0.15s; }` など軽微なトランジションはあるが、`prefers-reduced-motion` のハンドリングが全体に見当たらない。実害は小さいが、Phase 7 のチェックリスト項目として形式的に満たしておくべき。

## 6. Phase 6: プラットフォーム適応

- **iOS（Capacitor）**: Ionic `ios` モードの Large Title collapse や `ion-back-button` は正しく機能する構成。safe area は Ionic 標準コンポーネントに委譲されており妥当
- **Android（Capacitor）**: `md` モードのダークテーマ変数は用意されているが、`global.scss` のカスタムスタイル（`.hero-card`, `.chip-row` 等）は mode 非依存のため md でも同じ見た目になる。これは意図的な「独自ブランド優先」の判断として妥当（Android Material 規範への追従は本アプリのブランドと衝突するため、翻訳の必要なし）
- **モバイル Web / デスクトップ Web**: 前述の通りレスポンシブ考慮が薄い。`.hero-card` の `margin: 14px 16px` は画面幅が広がるほど間延びする。**提案**: `max-width: 640px; margin-inline: auto;` 相当をコンテンツラッパーに設定し、広い画面でも可読幅を保つ（HIG Layout の「layout guide による可読幅の制限」の Web 翻訳）

## 7. Phase 7: アクセシビリティ・品質監査

| チェック項目 | 現状 | 判定 |
|---|---|---|
| タップ領域 44×44px | `.chip-row ion-chip` は `height: 30px`、`ion-segment-button` は `min-height: 40px` — いずれも 44px 未満 | ⚠️ 要修正 |
| テキストコントラスト 4.5:1（17px未満） | 未実測。`.reading`(12px)・`.chip-on-color`(11px) 等は要検証 | ⚠️ 要検証 |
| Dynamic Type 相当（ブラウザのテキストサイズ拡大で崩れない） | インラインの固定 px + トークン未整備で未検証 | ⚠️ 要検証 |
| ライト/ダーク両対応 | `prefers-color-scheme` で実装済み、ios/md 双方に変数あり | ✅ |
| 色だけに依存しない情報伝達 | フィルターチップの選択状態は色のみ（アイコン/太字などの併用なし） | ⚠️ 要修正 |
| VoiceOver・フォーカス順序 | `aria-expanded` は探索トグルに付与済みだが、チップ・セグメント等への `aria-pressed`/`aria-label` の付与は未確認 | ⚠️ 要監査 |
| キーボード操作・フォーカスリング | `.exploration-toggle` にのみ `:focus-visible` あり、ボタン/チップ全般には未整備（Web 版で特に重要） | ⚠️ 要修正 |
| Reduce Motion 対応 | 未実装 | ⚠️ 要修正 |
| プライバシー（必要最小限のデータ要求） | 位置情報・通知等の許可要求は現状なし（該当なし） | ✅ |

## 8. 優先順位付き改善提案

### P0（一貫性・機能性の致命的な欠落 — 次回スプリントで着手）
1. souvenir-detail を sight-detail と同一のデザインパターン・機能セットに統一する（§4.1）
2. Discover の「寺社/庭園」フィルターチップを実装するか撤去する（§2.3）
3. DESIGN.md と実装（2つの同格ヒーローカード、消えた「別の提案」ボタン）を再同期する（§2.2, §4.2）

### P1（アクセシビリティ・一貫性の底上げ）
4. フィルターチップ・セグメントボタンのタップ領域を 44px 以上に拡大（§7）
5. フィルター選択状態を色以外（チェックアイコン等）でも表現（§7）
6. タイポグラフィをトークン化し、インラインスタイルを撤去（§3.1）
7. 削除操作に Undo 付きトーストを追加（§5.1）
8. フォーカスリング・`aria-*` 属性をインタラクティブ要素全般に拡充（Web 版の主要ユーザーはタッチだが、審査・デバッグ・将来のデスクトップ利用を考慮）

### P2（仕上げ・スケール対応）
9. コンテンツ幅にレスポンシブな上限を設定し、タブレット/デスクトップ Web での間延びを防ぐ（§6）
10. `prefers-reduced-motion` 対応（§5.3）
11. コントロール層（ツールバー/タブバー）とコンテンツ層（カード）のマテリアル的な区別を強める半透明処理（任意、§3.3）

## 9. やらないこと（禁則の確認）

- San Francisco/New York フォントや SF Symbols の移植はしない（現状も未使用で適合済み）
- Liquid Glass の見た目模写（過剰な `backdrop-filter`）は導入しない。導入するなら構造原理（層分離）目的に限定し、透明度は控えめにする
- 「Apple っぽい見た目」を目的化した意匠変更（角丸の変更、アイコンスタイルの変更等）は提案しない。現状の朱×和紙×明朝体のエディトリアル・アイデンティティは維持する
