# 構造・ナビゲーション・提示（Presentation）

出典: /modality /sheets /alerts /action-sheets /popovers /toolbars（2025-06 改訂で navigation bars を統合）/tab-bars /sidebars /split-views /searching /windows /panels /scroll-views

## ナビゲーション方式の選定

| 方式 | 使う場面 | 使わない場面 |
|---|---|---|
| タブバー（/tab-bars） | アプリのトップレベル区分間の移動。各区分のナビゲーション状態を保持 | アクション提供（それはツールバーの役割）。区分数は少ないほど良い。複雑な階層はサイドバー検討 |
| サイドバー（/sidebars） | 複雑な情報構造のトップレベル一覧（iPadOS/macOS/visionOS） | 浅い構造。iPadOS では convertible tab bar（サイドバー⇔タブバー切替可能）が選択肢 |
| スプリットビュー（/split-views） | 階層の複数レベルを同時表示（一覧→詳細）。2〜3 ペイン | 各ペインで現在選択を持続的にハイライトし、位置感覚を保つこと。2025-06 改訂で iOS/iPadOS 考慮事項追加 |
| 階層ナビゲーション（push） | 一本道の深掘り。ナビゲーションバー（現ツールバー統合）で戻り導線を常設 | — |
| モーダル | 下記 Modality の条件を満たすときのみ | 通常のナビゲーションの代替 |

- **タブバーは隠さない**（現在地を見失う）。例外はモーダルによる一時的被覆のみ
- ツールバー（2025-06 改訂で navigation bar を統合）: ビュー上端/下端の水平コントロール群。内容は「現在ビューのタイトル / 戻る・検索等のナビゲーション / アクション」の 3 種。**詰め込まない**。幅が縮むと macOS/iPadOS はシステムが自動でオーバーフローメニューを付ける（手動で付けない・デフォルトで溢れるレイアウトにしない）。優先度の低いアクションは More メニューへ

## Modality（/modality）— モーダルにして良い 4 条件

1. 重要情報を確実に受け取らせ、必要なら行動させる（アラート）
2. 直前の操作の確認・修正の選択肢を出す（アクションシート/確認ダイアログ）
3. 文脈を失わずに完了できる**狭い単一タスク**を実行させる（シート、ポップオーバー、別ウィンドウ）
4. 没入体験・集中を要する複合タスク（フルスクリーンモーダル）

モーダルは明示的な操作で閉じられること。多段モーダルの積み重ねは避ける。

## 提示コンポーネントの使い分け（頻出の判断）

| 状況 | 使う部品 |
|---|---|
| 予期しない問題・取り消せない破壊的操作の確認・購入確認 | **アラート**（/alerts）。ただし乱用禁止：情報提供だけのアラート、頻繁で取り消し可能な削除への確認は出さない。起動直後のアラートも避ける |
| ユーザーが自分で始めた操作に関連する複数の選択肢 | **アクションシート / 確認ダイアログ**（/action-sheets）。アラートとの違い＝「予期された文脈」で「追加の選択肢」を出す |
| 現在の文脈に密接な、スコープの狭いタスク（情報入力・保存先選択・書式指定） | **シート**（/sheets）。親ビューが一部見えるので文脈が保たれる。macOS/visionOS/watchOS では常にモーダル、iOS/iPadOS ではノンモーダルシートも可（例: Notes の書式シート） |
| 少量の情報・機能を一時的に出す（操作後すぐ消える） | **ポップオーバー**（/popovers）。矢印は起点要素を指し、起点や必要なコンテンツを覆わない。外側タップで閉じるのが基本。Close ボタンは保存/破棄の区別など明確化に役立つときだけ |
| 補助的な機能群を独立した場所に常駐 | **パネル**（macOS /panels）または visionOS の ornaments |
| 独立した文書・タスク | **別ウィンドウ**（iPadOS/macOS/visionOS /windows） |

判別のコツ: 「予期せぬ知らせ」→アラート、「自分の操作の枝分かれ」→アクションシート、「小タスクの作業台」→シート、「ちょい見せ」→ポップオーバー。

## 検索（/searching, /search-fields）

- 検索が主要機能なら**一次的な操作**として配置する（タブや常設検索フィールド）。2025-06 改訂で search fields の一般指針を統合
- 最近の検索・サジェスト・補完・修正で入力負荷を下げる。スコープコントロールやトークンで絞り込みを支援
- アプリ内検索（検索フィールド）、文書内検索（find）、システム横断（Spotlight 連携＝コンテンツをインデックス提供）の 3 層を区別する

## スクロールビュー（/scroll-views）

- スクロール可能なことを示唆する（部分表示・インジケータ）。ページング動作は明確な単位があるときのみ
- コンテンツはビューの端（下端・側端）まで伸ばし、コントロール層は scroll edge effect で分離する（foundations.md の Layout 参照）

## macOS ウィンドウ（/windows, /panels, /the-menu-bar）

- ウィンドウ: タイトルバー・ツールバー統合の作法、状態復元、リサイズへの適応。重要コントロールを下端に置かない
- メニューバー: アプリの**全機能をメニューバーから到達可能に**するのが macOS の作法。標準メニュー（App/File/Edit/View/Window/Help）の構成と標準ショートカットを尊重する（詳細は原典 /the-menu-bar）
